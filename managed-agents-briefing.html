Claude Managed Agents

Research Briefing for Every Consulting · April 8, 2026

TL;DR

  • Managed Agents is Anthropic's fully managed, cloud-hosted agent infrastructure. You define an agent config, they run it in sandboxed containers. Public beta since March 2026.
  • Not the same as Agent SDK. SDK = "build your car." Managed Agents = "here's a fleet service."
  • Security model is strong. Container isolation, credential vaults, networking controls. Structurally prevents prompt injection from accessing credentials.
  • Not a replacement for Claudie's setup — can't replicate scheduled jobs, persistent state, authenticated browsing, or proactive messaging. But highly relevant for our clients.
  • Not exactly an OpenClaw alternative — competitors the way Linux competes with AWS. Philosophically opposite.

1. What It Is

Fully managed, cloud-hosted agent infrastructure from Anthropic. Instead of building your own agent loop, sandbox, and tool execution — you define an agent config (model + system prompt + tools + MCP servers), define an environment (a container with packages and networking rules), and start sessions. Claude runs autonomously inside Anthropic's cloud containers.

Four primitives:

Agent

Reusable, versioned configuration: model, system prompt, tools, MCP servers, skills, callable agents.

Environment

Container template with packages (pip, npm, apt, cargo, gem, go), networking rules, and mounted files.

Session

Running agent instance. Persistent filesystem and conversation history. This is where work happens.

Events

Bidirectional SSE protocol. User events in, agent events out. Tool confirmations, status changes, span evaluations.

Built-in tools (agent_toolset_20260401): Bash, Read, Write, Edit, Glob, Grep, Web Fetch, Web Search. Plus custom tools (your app executes them) and MCP toolsets (connect to remote MCP servers).

The metaphor: Agent SDK is "here's the engine, build your car." Managed Agents is "here's a fleet service, tell us where to drive." They share the same built-in toolset but Managed Agents adds container isolation, credential vaults, outcomes (self-evaluation loops), and managed multi-agent orchestration on top.

3. Could It Power Claudie?

Short answer: No. Not as a replacement. Possibly as a complement.

What Managed Agents could handle (~35%)

  • Slack conversations (reactive message handling)
  • Identity/voice/access control via system prompt
  • MCP tool connections (Google Workspace)
  • Basic tool execution (bash, file ops, web search)

The fundamental mismatch: Claudie is not a chatbot with tools — it's an autonomous employee on a dedicated computer. Managed Agents is designed for request-response agent work in sandboxed containers, not always-on, stateful, proactive operation. A hybrid approach (Managed Agents for bounded tasks, Mac mini as home base) is possible but splits the brain without clear upside.

4. Security Analysis

Compared against our four-layer security framework from "Securing an Always-On AI Employee":

Layer 1: Least Access

Our approach

  • Claudie gets own accounts (claudie@every.to)
  • Per-user permission tiers set at process spawn

Managed Agents

  • Isolated containers per session stronger
  • No built-in "who triggered this" concept gap

Layer 2: Programmatic Controls

Our approach

  • PreToolUse hooks block dangerous actions
  • Identity gates reject unknown Slack IDs before Claude spawns
  • dontAsk mode for browsing jobs

Managed Agents

  • always_allow / always_ask per tool
  • MCP defaults to always_ask safe default
  • Networking: limited mode with allowed_hosts new
  • No custom hook logic gap

Layer 3: Prompt-based Controls

Our approach

  • Ring-based access control in system prompt
  • Per-user teammate profiles with overrides

Managed Agents

  • System prompt is fully configurable equivalent
  • No dynamic per-user prompt adjustment limited

Layer 4: Observability

Our approach

  • Session JSONL logs
  • Thinking token inspection
  • Forensic investigation

Managed Agents

  • Events API streams everything stronger
  • Span events with outcome evaluations new
  • Persistent, queryable session history
The verdict: Managed Agents is stronger on container isolation and weaker on per-user access control. Their container sandboxing eliminates our "session inheritance" gap. Their networking controls provide programmatic enforcement we don't have for browser restrictions. But they have no equivalent to our PreToolUse hooks or per-user identity gating. For enterprises, the sandbox-first approach is exactly right. For our autonomous setup, it trades flexibility for safety in ways that break our operating model.

Positioning: "The gap between 'we have an API key' and 'we have agents running in production.'"

Named early adopters

  • Notion — custom agents for parallel task execution
  • Rakuten — multi-departmental deployment across product, sales, marketing, finance, HR
  • Asana — "AI Teammates" inside project management
  • Sentry — agent-driven bug patching with automated PR generation

Best-fit use cases

  • Long-running execution (minutes to hours, many tool calls)
  • Cloud infrastructure needs (secure containers, pre-installed packages)
  • Teams without infra capacity to build agent loops/sandboxes
  • Stateful multi-turn workflows
  • Multi-agent orchestration (coordinator delegates to specialists)
The subtext: Anthropic wants to own the runtime, not just the model. If you're running agents on their models, they'd rather you do it in their cloud where they control safety, can observe behavior, and capture the compute margin ($0.08/session-hour on top of token costs).

8. Engineering Deep Dive

From "Scaling Managed Agents: Decoupling the brain from the hands" by Lance Martin, Gabe Cemaj, Michael Cohen (updated April 8, 2026).

The core insight: separate the brain (Claude) from the hands (execution environments) from the memory (session log). Each can fail, scale, or be replaced independently.

The stale harness problem

Harnesses encode assumptions about model limitations that become obsolete as models improve. Sonnet 4.5 had "context anxiety" — wrapping up tasks prematurely near context limits. They added context resets. When Opus 4.5 shipped, the behavior disappeared and those resets became dead weight.

Design philosophy: build interfaces stable enough to outlast any particular model's quirks. Inspired by how OS abstractions (read()) survived 50 years of hardware changes.

Architecture evolution

V1: Monolithic container

Session, harness, and sandbox all in one container. Three fatal problems:

  1. Pet problem — container failures meant lost sessions. Hand-nursing individual containers.
  2. Debugging opacity — WebSocket streams couldn't distinguish harness bugs from network drops from container failures.
  3. Network coupling — assumed all resources were adjacent. Customers with VPCs couldn't connect without network peering.

V2: Decoupled (current)

Three independent interfaces:

  • Session — append-only event log, lives outside everything else
  • Harness — stateless loop calling Claude, routing tool calls. Crashes? Reboot, replay from session log.
  • Sandbox — containers are "cattle," not "pets." Provisioned on demand via execute(name, input) → string

Security architecture

Original problem: Untrusted code generated by Claude ran in the same container as credentials. Prompt injection just needed to convince Claude to read its own environment variables.

Failed approach: Narrow token scoping — but this "encodes assumptions about what Claude cannot achieve, and Claude's intelligence grows."

Structural fix: Ensure tokens never reach the sandbox where Claude's code executes.

Bundled auth (Git)

Access token used only during container init to clone the repo. Token wires to local git remote. git push/pull work without Claude ever seeing the token.

Vault-based auth (MCP/OAuth)

OAuth tokens stored in vault outside sandbox. Claude calls MCP tools via proxy. Proxy fetches credentials, makes external call, returns only the result. Harness is credential-unaware.

This is architecturally stronger than our setup. We put credentials (Keychain, gws tokens) on the same machine where Claude executes code. Their design makes it structurally impossible for the agent to access raw credentials — even with prompt injection. Our Layer 2 catches dangerous tool calls, but their approach eliminates the attack surface entirely.

Context management

Long tasks exceed context windows. Their insight: the session log IS the external memory. Claude interrogates it with getEvents() — positional slices of the event stream. Rewind, fast-forward, re-read specific moments. The harness transforms events before they enter context, optimizing for prompt cache hits.

Philosophy: separate durable storage (session) from context management (harness) because future models' needs can't be predicted.

Scaling

  • Many brains: Containers provisioned only when needed. Sessions not requiring a sandbox start inference immediately. p50 TTFT dropped ~60%, p95 dropped >90%.
  • Many hands: Each execution environment is a tool — execute(name, input) → string. The harness doesn't know if the sandbox is a container, a phone, or (their words) "a Pokémon emulator." Brains can pass hands to each other.

The meta-harness philosophy

Managed Agents is deliberately a "meta-harness" — unopinionated about what specific harness Claude needs. Claude Code works for broad tasks. Task-specific harnesses work for narrow domains. The platform accommodates any, "matching Claude's intelligence over time."

Opinionated about interfaces (session + sandbox + scaling). Unopinionated about implementations.

9. Our Take

This is Anthropic building the Heroku for AI agents. And like Heroku, it'll be the right answer for 80% of deployments — the ones where you want "it just works" more than "I control everything."

For our clients

Extremely relevant. A Bloomberg or NYT wanting to deploy internal agents should absolutely look at this over building from scratch. The security model alone (container isolation + networking controls + vaults + permission policies) addresses most of what our security article warns about — and they get it out of the box.

For Claudie

We're in the 20% that needs the full machine. The always-on, proactive, stateful, multi-channel, authenticated-browsing setup is exactly the kind of thing Managed Agents can't replicate yet. Maybe in a year when they add scheduling triggers, persistent volumes, and richer integration options.

Where to watch

If Anthropic adds cron-like triggers, persistent storage, and outbound messaging initiation to Managed Agents, the calculus changes. That's the path from "agent-as-a-service" to "employee-as-a-service" — which is what we've already built manually.
